# Тестовое задание: порт экрана регистрации с C/Informix на Python

## Контекст

Старая университетская система написана на **C со встроенным SQL (Informix ESQL/C)**. Мы переносим её экраны на Python (frappe framework). В репозитории — один экран целиком, со всеми слоями. Он выполняет три операции над регистрацией студента в семестре:

- **drop** — снять студента с секции;
- **add** — записать на секцию (проверки: hold, окно регистрации, пререквизиты,
  кредитный лимит, места, лист ожидания);
- **swap** — заменить одну секцию на другую (drop, затем add).

```
legacy/src/   main.c (драйвер) proc.ec (бизнес-логика) db.ec (доступ к БД) dec.h (коды/константы)
legacy/sql/   schema.informix.sql   — оригинальная схема (для справки)
data/         schema.sqlite.sql + seed.sqlite.sql — ваш порт работает с этим
```

Компилировать `.ec` не нужно — это спецификация. Как читать ESQL/C: обычный C, но строки с `$` (`$select`, `$insert`, `$update`, `$delete`, `$COMMIT/ROLLBACK WORK`) — встроенный SQL;
 `$var` в запросе — host-переменная (в Python — параметр `?`); 
 `$char`/`$long` — типы; результат проверяется через `SQLCODE` (`100`/`SQLNOTFOUND` — строк нет, `<0` — ошибка БД, `0` — успех). 

## Что нужно сделать

Рабочий end-to-end порт экрана на Python, **поведенчески эквивалентный оригиналу**.

**Рамки:** Python 3.10+, только стандартная библиотека, БД — `sqlite3` поверх
`data/*.sqlite.sql`. Без внешних зависимостей (`pytest` допустим как dev-зависимость для тестов). Структуру кода выбираете сами.

**Сохраните разделение на три слоя:** доступ к данным (`db.ec`, один запрос на функцию, без бизнес-правил) · бизнес-логика (`do_drop`/`do_add`/`do_swap`, правила и **их порядок**, коды исхода) · тонкий вход (`main.c`, разбор аргументов, печать сообщения, exit-код).

### Обязательно

1. Операции **drop** и **add**, три слоя.
2. CLI-контракт (см. ниже): **дословные** сообщения и exit-коды.
3. Транзакции: при ошибке БД — **полный откат, без частичных изменений**, успех фиксируется.
4. Прохождение сценариев приёмки **S1–S13**.

### На выбор

- **A. `swap`** (сценарий S14).
- **A+. Атомарный `swap`**: вся операция в одной транзакции, при любом сбое — полный откат (S15). 
- **B. Автотесты** — основной способ показать корректность. Идеально — покрытие всех сценариев.

## CLI-контракт

```
<entrypoint> drop <id> <sec> <yr> <sess>
<entrypoint> add  <id> <sec> <yr> <sess>
<entrypoint> swap <id> <dropsec> <addsec> <yr> <sess>
```
